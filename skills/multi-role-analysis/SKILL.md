# 多角色分析技能

从多个专家角色视角对问题进行深入分析，通过**角色间的深度辩论**揭示单一视角容易忽略的关键因素。

## 核心原理

**真正的洞察来自冲突，而非共识。** 每个角色看到的是真相的一个切面，但真正的价值在于这些切面碰撞时产生的火花。

```
用户输入 ──→ 议题拆解 ──→ 辩论设计 ──→ 多轮辩论 ──→ 妥协记录 ──→ 决策输出
                │              │            │            │
                ↓              ↓            ↓            ↓
           [关键决策点]    [灰色地带]    [回合制]    [谁让步了]
```

---

## 消息完整性检测

在开始分析前，检测用户消息是否完整：

### 完整性判断规则

```
消息以以下符号结束 → 视为完整：
- 中文句号 。
- 英文句号 .
- 问号 ？?
- 感叹号 ！!
- 结束标记 [END]、完成、完毕

消息以以下情况结束 → 视为可能不完整：
- 逗号 ，,
- 省略号 ...
- 突然中断（无标点）
- 明显的语法不完整
```

### 不完整消息处理

如果检测到消息可能不完整：

1. **温和提示**：简要说明检测到消息可能未完成
2. **提供选项**：
   - 继续发送当前内容进行分析
   - 补充内容后再发送
3. **不阻塞**：用户可以选择继续，不应强制等待

## 🎯 深度辩论设计原则

### 原则 1：设计灰色地带议题

**问题**：角色立场差异太大，反而容易快速达成一致。

**解决方案**：选择每个角色都有合理关切的议题，而非简单对立。

```
❌ 不好的议题设计：
- 产品经理：功能越多越好
- 技术架构师：功能越少越好
→ 太容易得出"折中"结论，没有深度辩论

✅ 好的议题设计：
- 产品经理：用 Vue 3（团队熟悉，开发快）
- 技术架构师：用 React（图表库生态更好）
- 项目经理：两者都行，但招人呢？
→ 每个角色都有合理关切，需要多轮辩论才能达成共识
```

### 原则 2：角色关切重叠

每个议题至少有两个角色有交集关切：

| 议题 | 产品经理 | 技术架构师 | 项目经理 |
|------|----------|------------|----------|
| 前端框架 | 开发速度、团队熟悉度 | 图表库生态、TypeScript支持 | 招人难度、维护成本 |
| 数据库选择 | 用户体验（性能） | 技术债务、扩展性 | 部署复杂度、测试工作量 |
| 实时推送 | 用户感知延迟 | 实现复杂度 | 资源消耗、服务器成本 |

### 原则 3：多轮辩论结构

```
第一轮：立论
├── 角色A陈述观点 + 理由
├── 角色B陈述观点 + 理由
└── 角色C陈述观点 + 理由

第二轮：质疑与反驳
├── 角色A质疑角色B的观点
├── 角色B质疑角色C的观点
└── 角色C质疑角色A的观点

第三轮：妥协与调整
├── 各角色评估对方观点的合理性
├── 提出折中方案或条件
└── 达成共识或记录分歧

第四轮：最终决策
├── 总结辩论结果
├── 记录谁妥协了、妥协条件是什么
└── 明确下一步行动
```

### 原则 4：量化权衡

用数字支撑辩论，而非模糊的"好/坏"：

```markdown
❌ 模糊辩论：
产品经理：Vue 开发快
技术架构师：React 图表库多

✅ 量化辩论：
产品经理：Vue 团队熟悉，前端框架 6h 可缩短到 4h
技术架构师：但 React 的 echarts-for-react 开箱即用，省去 2h 封装时间
项目经理：两者算下来时间差不多，但 React 招人更容易
```

## 辩论流程

### Phase 1: 议题拆解

从输入中识别需要辩论的关键决策点：

```markdown
## 议题识别

1. **议题一**：[决策点描述]
   - 涉及角色：产品经理、技术架构师、项目经理
   - 灰色地带：[每个角色都有合理关切的地方]

2. **议题二**：[决策点描述]
   - 涉及角色：技术架构师、安全专家
   - 灰色地带：[...]
```

### Phase 2: 多轮辩论

**格式要求**：使用对话形式，展示角色间的互动

```markdown
## 🗣️ 辩论过程

### 第一轮：[议题名称]

**产品经理**：[观点] + [理由]

**技术架构师**：⚠️ [质疑/反对] + [理由]

**项目经理**：[补充视角/支持/折中]

---

**产品经理**：[回应质疑/调整观点]

**技术架构师**：[进一步说明/提出条件]

**项目经理**：✅ [总结/提出折中方案]

---

### 辩论结论

| 最终决策 | 妥协方 | 妥协条件 |
|----------|--------|----------|
| [决策内容] | [谁让步了] | [让步的条件是什么] |
```

### Phase 3: 综合结论

```markdown
## 📋 最终共识

| 议题 | 决策 | 妥协方 | 妥协条件 |
|------|------|--------|----------|
| 前端框架 | React | 技术架构师 | 先做技术调研确认图表库差异 |
| 数据库 | SQLite | 技术架构师 | 预留数据库抽象层方便迁移 |
| 实时推送 | 轮询 → WebSocket | 产品经理 | v1.5 支持WebSocket |

## ⚠️ 仍待讨论

1. [未解决的议题]
2. [需要进一步调研的问题]
```

## 辩论技巧

### 1. 反对时先承认合理性

```
❌ 错误示例：
技术架构师：你说的不对，Vue 生态很差。

✅ 正确示例：
技术架构师：Vue 确实学习曲线低，团队上手快。但我想问一个问题：如果未来要对接专业金融图表库，Vue 的选择就比较有限了。
```

### 2. 用问题引导思考

```
❌ 错误示例：
项目经理：WebSocket 太复杂了，不要用。

✅ 正确示例：
项目经理：我有一个顾虑——如果我们用 WebSocket，服务器资源消耗会增加多少？用户量达到多少时需要扩容？
```

### 3. 提出条件而非绝对反对

```
❌ 错误示例：
产品经理：我不同意用 SQLite。

✅ 正确示例：
产品经理：SQLite 性能确实够用，但我有一个条件：架构上要预留数据库抽象层，未来如果用户数据量增长，可以平滑迁移到 PostgreSQL。
```

### 4. 记录"谁背锅"

```
**产品经理**：如果用户抱怨数据延迟，这个锅谁背？
**技术架构师**：我背。架构上会设计数据推送抽象层，方便切换到 WebSocket。
```

## 注意事项

1. **设计灰色地带**：每个议题至少有两个角色有交集关切
2. **多轮辩论**：不要一轮就结束，要有质疑、反驳、妥协的过程
3. **量化权衡**：用时间/成本/性能的数字支撑辩论
4. **记录妥协**：明确谁让步了、让步条件是什么
5. **承认不确定性**：当信息不足时，记录为"待调研"
6. **保持角色一致性**：每个角色有明确的思维模式和利益取向

## 示例

### 输入

```
我们打算做一个投资分析系统，需要考虑技术选型。
```

### 输出

```markdown
# 多角色深度辩论报告

## 分析概要
| 项目 | 内容 |
|------|------|
| **问题** | 投资分析系统技术选型 |
| **角色** | 产品经理、技术架构师、项目经理 |
| **议题数** | 3 |

---

## 🗣️ 辩论过程

### 第一轮：前端框架选择

**产品经理**：前端我建议用 Vue 3 + Vite。学习曲线低，开发速度快。团队成员之前用过 Vue，上手快。

**技术架构师**：我有些顾虑。Vue 确实学习曲线低，但 PRD 里写的是 React + TypeScript。ECharts 的 React 封装更成熟。如果换 Vue，可能需要额外封装工作。

**项目经理**：从时间角度看，Vue 确实更快。但问题是——我们确定未来不会招 React 开发者吗？React 市场份额更大，招人更容易。

**产品经理**：但这是内部工具，不是商业项目。团队稳定性更重要。用熟悉的技术栈，开发效率至少提升30%。

**技术架构师**：...你说得有道理。但我担心的是另一个问题：如果未来要对接第三方金融组件库，React 的生态更好。比如 React-Stockcharts 这些库，Vue 的选择就少很多。

**项目经理**：要不这样——技术架构师你评估一下图表组件库的差异，如果 React 确实有明显优势，就用 React；如果差不多，就用团队熟悉的 Vue？

**技术架构师**：好，我来做一个技术调研。1小时出结果。

---

### 第二轮：后端数据存储

**技术架构师**：后端存储我建议用 PostgreSQL。金融数据需要复杂查询，SQLite 的并发能力太弱。

**项目经理**：⚠️ 等等。PRD 明确写的是 SQLite。部署简单，用户本地安装就能用，不需要额外配置数据库服务。这降低了用户门槛。

**产品经理**：我同意项目经理。这是桌面应用，用户可能不懂数据库。SQLite 开箱即用，零配置。

**技术架构师**：但你们没考虑性能问题。策略回测需要处理大量历史数据，SQLite 的并发读写性能会很糟糕。用户回测一个策略，界面卡死10秒，这是可以接受的吗？

**产品经理**：...你说得有道理。但我们能不能用其他方式优化？比如异步任务、结果缓存？

**项目经理**：技术上可以，但开发工作量会增加。而且如果用户数据量增长到几百万条，SQLite 的性能瓶颈还是会显现。

**技术架构师**：那我提一个折中方案：
- 默认用 SQLite，满足大部分用户需求
- 提供 PostgreSQL 迁移选项，给高级用户使用

**产品经理**：这样会增加维护成本吧？两套数据库适配...

**项目经理**：是的。而且测试工作量翻倍。我建议还是做选择：
- 要么 SQLite + 接受性能限制 + 简化部署
- 要么 PostgreSQL + 性能保障 + 增加部署复杂度

**技术架构师**：...好吧。我重新评估一下。其实对于个人用户，数据量应该不会超过10万条。SQLite 加上索引优化，性能应该够用。

**产品经理**：那就先 SQLite？等用户反馈性能问题再迁移？

**技术架构师**：✅ 同意。但架构上要预留数据库抽象层，方便未来迁移。

---

### 第三轮：实时数据推送方案

**技术架构师**：实时行情推送，我建议用 WebSocket。延迟低，服务器主动推送，用户体验好。

**产品经理**：WebSocket 好！用户看到的是实时数据，这才是专业投资工具该有的体验。

**项目经理**：⚠️ 我有不同意见。WebSocket 实现复杂：
1. 需要维护长连接，服务器资源消耗大
2. 断线重连逻辑复杂
3. 部署时需要支持 WebSocket 的服务器

如果只是个人工具，用轮询就够了。每5秒请求一次，延迟可以接受。

**产品经理**：但5秒延迟对股票交易来说很长了。用户可能错过最佳交易时机。

**技术架构师**：支持产品经理。金融数据实时性很重要。但项目经理说的技术复杂度也是问题...

**项目经理**：那我提一个问题：我们有多少用户？如果只有几十个，轮询完全够用；如果是几千个，WebSocket 才有优势。

**产品经理**：初期用户应该不多...但我们不能限制用户规模啊。

**技术架构师**：要不这样：
- MVP 阶段用轮询（5秒间隔）
- v1.5 阶段支持 WebSocket（用户可选）

这样初期开发简单，后续迭代优化。

**项目经理**：可以接受。但我要强调：轮询方案的前端代码也要预留 WebSocket 接口，避免后期大改。

**产品经理**：...行吧。但用户如果抱怨数据延迟，这个锅谁背？

**技术架构师**：我背。架构上会设计数据推送抽象层，方便切换。
```
