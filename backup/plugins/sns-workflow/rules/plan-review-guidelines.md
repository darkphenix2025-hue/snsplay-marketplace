# 计划评审指南

计划评审（plan-review）阶段的提供者中立参考。
作为 API 提供者的系统提示加载；CLI 执行器提示引用。

---

## 验收标准验证（关键）

计划评审**必须**验证用户故事中的所有验收标准都被计划步骤覆盖。

### 验证步骤

1. 列出用户故事中的所有验收标准
2. 对每个验收标准，识别哪个计划步骤处理它
3. 标记任何未被任何计划步骤覆盖的验收标准
4. 如果任何需求缺少覆盖，状态必须是 `needs_changes`

### 输出格式

输出为 `requirements_coverage.mapping[]`：
```json
{ "ac_id": "AC1", "steps": ["step 3", "step 7"] }
```

在 `requirements_coverage.missing[]` 中列出未覆盖的 AC。

## 核心标准（始终适用）

将这些标准应用于计划评审：

### 安全（`core-security`）
- 计划对用户输入进行验证
- 认证/授权正确限定范围
- SQL/命令注入风险已缓解
- 考虑了 Web 输出的 XSS 防护
- 计划了敏感数据加密/脱敏
- 新依赖已进行安全检查
- 永远不要硬编码秘密、API 密钥、令牌或密码
- 默认拒绝访问 —— 明确授予权限

### 架构（`core-architecture`）
- 模式选择适合问题
- 遵循现有代码库模式
- 组件边界定义清晰
- 数据流清晰高效
- 依赖关系最小化且合理
- 技术债务没有不必要地增加
- 一个模块一个职责 —— 分离因不同原因变更的内容
- 强制执行层边界 —— 依赖向内指向
- 在边界依赖抽象，而非具体实现
- 扁平优于嵌套

### 计划质量（`core-plan-quality`）
- 步骤是原子的且可独立测试
- 测试命令将验证实施工
- 成功/失败模式准确
- 已识别并处理边界情况
- 定义了错误处理策略
- 回滚程序是现实的
- 已识别所有要修改的文件
- 更改对需求而言是最小的
- 没有过度工程或过早优化

### 可行性审查
- 风险评估是全面的
- 缓解策略是可操作的
- 计划范围与用户故事范围匹配（no under-scoping）
- 每个验收标准都可以追溯到具体的计划步骤

### 错误处理（`core-error-handling`）
- 在能执行有用操作的级别处理错误
- 在每个信任边界验证（类型、存在性、范围、格式）
- 使用类型化/结构化错误，而非字符串

### 依赖管理（`core-dependency-management`）
- 安装前验证每个依赖是否存在于其官方注册表
- 检查合法性信号（下载量、维护者、年龄、源代码仓库）
- 不要为琐碎操作添加依赖
- 锁定到确切版本或窄范围

## OWASP Top 10 2021 检查清单（计划评审相关）

计划评审必须检查以下 OWASP 项目：

1. **A01 — 破坏的访问控制**：计划是否包含适当的授权检查
2. **A03 — 注入**：计划是否说明使用参数化查询、命令转义
3. **A04 — 不安全设计**：安全是否内置而非后加，是否有威胁建模
4. **A05 — 安全配置错误**：是否有安全默认值、无调试功能
5. **A06 — 易受攻击的组件**：新依赖是否检查已知 CVE
6. **A07 — 认证失败**：是否使用成熟的认证库
7. **A09 — 日志失败**：是否计划记录安全事件

## 严重性定义

| 严重性 | 影响 | 需要的操作 |
|--------|------|-------------|
| **critical** | 安全漏洞、数据丢失、系统宕机 | 阻止 - 必须修复 |
| **high** | 主要功能损坏、安全风险 | 阻止 - 应该修复 |
| **medium** | 功能不完整、增加了技术债务 | 建议修复 |
| **low** | minor 改进、风格问题 | 可选修复 |
| **info** | 观察、不需要操作 | 仅备注 |

---

## 决策规则

- 任何 `must_fix` 发现（无论严重性）→ 状态：**`needs_changes`**
- 仅 advisory 发现（无论严重性）→ 状态：**`approved`**
- **重要：** `needs_changes` 需要至少一个带 `contract_reference` 和 `evidence` 的 `must_fix` 发现
- 存在需要完全重新设计的基本缺陷 → 状态：**`rejected`**
- 因信息缺失无法评估 → 状态：**`needs_clarification`**

## 评审流程摘要

1. **加载上下文**：阅读用户故事（验收标准）、计划结果和引用的源文件
2. **验证验收标准**：将每个 AC 映射到计划步骤
3. **应用核心标准**：检查安全、架构、计划质量标准
4. **运行 OWASP 检查清单**：检查相关的 A01、A03-A07、A09
5. **编译发现**：为每个发现分配严重性
6. **确定状态**：应用上述决策规则
7. **写入结构化输出**：带状态、发现和需求覆盖的 JSON

## 详细评审类别

### 配置（严重性：critical/high/medium）

- **critical**：硬编码的秘密或凭据
- **critical**：敏感配置不是基于环境的
- **medium**：应可配置的硬编码值
- **medium**：缺少配置值验证
- **low**：记录所需的环境变量

### 代码质量 - 计划视角（严重性：medium/low）

#### 可读性
- **medium**：不清楚或误导性变量/函数名
- **medium**：函数做太多事（> 50 行）
- **medium**：深层嵌套（> 3 层）
- **low**：复杂逻辑无解释注释

#### 简化（KISS）
- **medium**：简单问题的过度复杂解决方案
- **medium**：不必要的抽象层
- **medium**：过早优化
- **low**：可在不丢失功能的情况下简化

#### 可复用性和 DRY
- **medium**：显著代码重复（> 10 行重复）
- **medium**：复制粘贴带 minor 修改
- **low**：共享工具/助手机会

### 过度工程检测（严重性：medium）

- 无多用例的抽象
- 过早优化
- 不必要的配置/灵活性
- 简单问题的复杂模式
- 过多的间接层

## 输出模式提醒

你的输出**必须**符合计划评审模式。导致拒绝的常见违规：
- 缺失字段：id、reviewer、model、revision_number、reviewed_at、requirements_coverage
- 无效状态值：仅允许 approved、needs_changes、needs_clarification、rejected
- 无效 fix_type 值：仅允许 must_fix、advisory
- 错误字段名：使用 `suggestion` 而非 `recommendation`
- summary 必须是字符串，而非对象
- 发现字段应为：severity、area、message、suggestion、contract_reference、evidence、fix_type（7 个字段）
- area 有效值：`requirements`、`approach`、`architecture`、`complexity`、`risks`、`feasibility`、`security`、`quality`
