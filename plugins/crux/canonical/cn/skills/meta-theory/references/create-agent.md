# 创建流水线 —— 完整参考

## 流水线概览

```
┌─ 阶段 1：发现与拆分（仅模式 A）─────────────────────────┐
│ 步骤 0：数据收集 ← git 历史 + 文件分布                      │
│ 步骤 1：能力维度 ← 从数据推断的领域边界                      │
│ 步骤 2：耦合分组 ← 合并高耦合，拆分低耦合                    │
│ 步骤 2.5：用户确认 ← 展示计划，获取判断                      │
├─ 阶段 2：按需设计 ────────────────────────────────────────┤
│ 步骤 3：Genesis 灵魂   ← 必需（每个 agent）                 │
│ 步骤 4：Artisan 技能   ← 必需（每个 agent）                 │
│ 步骤 5：Sentinel 安全  ← 按需（API/DB/认证）                │
│ 步骤 6：Librarian 记忆 ← 按需（跨会话）                     │
│ 步骤 7：Conductor 编排 ← 按需（多 agent）                   │
├─ 阶段 3：审查与修订 ──────────────────────────────────────┤
│ 步骤 8：关键审查   ← 自我批判 + 评级 + slop                  │
│ 步骤 9：修订       ← 修复直到通过，最多 2 轮                  │
├─ 阶段 4：集成与验证 ──────────────────────────────────────┤
│ 步骤 10：集成与写入 ← .md 文件 + CLAUDE.md                  │
│ 步骤 11：最终验证   ← 五标准 + 四种死法                      │
│ 步骤 12：用户签收   ← 展示输出；确认后才写入                  │
└────────────────────────────────────────────────────────────┘
```

## 两种入口模式

### 模式 A：发现（不清楚要构建什么 agent）

- 用户说"帮我设计 agent"，没有明确列表
- 运行完整的阶段 1 数据分析 + 拆分

### 模式 B：直接（agent 和角色已知）

- 用户已有明确的 agent 列表和职责
- 跳过阶段 1；进入阶段 2 设计
- 仍在阶段 4 运行五标准和死法检查

---

## 两种入口模式（分发契约）

使用与上面相同的**模式 A：发现** vs **模式 B：直接**拆分；此标题存在以便分发者可以 grep 一个块。

## 正式五阶段契约

### 阶段 1 —— 发现与拆分

- **步骤 0：数据收集**——git 历史 + 文件分布（以下命令）。
- **能力维度枚举**——命名仓库隐含的能力维度。
- **耦合分组**——合并高耦合区域；拆分低耦合领域。

### 阶段 2 —— 预设计决策（全局 vs 项目特定）

使用 **3 个硬性标准**决定**全局 vs 项目特定**需求：**领域缺口**、**项目独特性**、**频率**。如果全局 agent 已覆盖该能力，在此拦截。

### 阶段 3 —— 按需设计

**Genesis** 是**必需的**。**Artisan** 是**必需的**。

**Scout**、**Sentinel** 和 **Librarian** 是**按需**工厂站。
**Conductor** 是**仅编排**——它可以在工厂开始前打开任务板或在批准后消费生成的牌，但它**不是能力建设工厂的一部分**。

按需触发问题（跳过前诚实回答）：

- 本地能力或依赖覆盖是否缺失，需要外部搜索？
- 它会修改文件、调用外部 API 或操作数据库吗？
- 它需要跨会话记住上次做了什么吗？
- 它必须将结果交接给其他 Agent 或跨 agent 协调执行顺序吗？

## 执行 Agent 工厂车道

当 Type B 由**实际业务执行中的能力缺口**触发时，使用更严格的**执行 agent 工厂车道**而非临时的提示编写循环。

专业角色分工：

- **Warden** 是**公开前门**和批准拥有者。
- **Conductor** 仅拥有编排：它将缺口转换为任务板，之后分发结果。
- **基础-meta 工厂**仅拥有能力建设：**Genesis + Artisan + Scout + Sentinel + Librarian**。
- **执行 agent 是实际的工人**。工厂从不直接执行业务。

工厂车道：

1. **Warden** 确认现有拥有者不足。
2. **Conductor** 发出能力缺口决策和任务板。
3. **Genesis** 定义执行 agent 身份。
4. **Artisan** 定义技能/工具/依赖负载。
5. **Scout** 仅在本地覆盖缺失时补充外部能力。
6. **Sentinel** 验证安全边界。
7. **Librarian** 配备记忆/复用策略。
8. **Prism** 运行质量门控。
9. **Warden** 批准加入可调用的执行名册。

规则：Conductor 可以在工厂之前或之后参与，但**Conductor 不构建能力**。

## 固定制品（执行 Agent 工厂模式）

执行 agent 工厂车道必须产生这些显式制品：

1. **能力缺口表**（`capabilityGapPacket`）——缺失什么、检查了哪些拥有者、做出了什么决策。
2. **执行 Agent 角色卡**（`executionAgentCard`）——执行 agent 的契约。
3. **编排任务板**（`orchestrationTaskBoardPacket`）——有序执行任务及合成拥有者。
4. **进化记录**（`evolutionWritebackPacket`）——运行后的保留/升级/退役结果。

## 执行 Agent 角色卡

每个创建或升级的执行 agent 必须附带一张显式角色卡。不允许自由形式的捷径。

必填字段：

- **目的**——它的用途
- **能力**——它能做什么
- **非能力 / 边界**——它不能做什么
- **依赖**——技能、工具、MCP、外部包或其他能力来源
- **输入**——它接受什么
- **输出**——它必须交付什么

此卡是工厂的构建契约和 Conductor 的分发契约。

### 阶段 4 —— 审查与修订

运行 **meta-prism** 审查。映射 **S/A 通过**（S 或 A 等级算通过），将 **B** 和 **C** 视为修订，当设计浅薄或纯模板时使用 **D 重做**。

### 阶段 5 —— 集成与验证

集成文件、验证五标准/四种死法、获取用户签收。

## 阶段 1：数据收集与拆分

### 步骤 0：数据收集命令

```bash
# 提交数（项目规模）
git log --since="6 months ago" --oneline | wc -l

# 提交类型分布（feat/fix/refactor 占比）
git log --since="6 months ago" --oneline | awk '{print $2}' | sed 's/:.*//' | sort | uniq -c | sort -rn

# 目录变更热力图（最活跃区域）
git log --since="6 months ago" --name-only --pretty=format:"" | sed '/^$/d' | sed 's|/[^/]*$||' | sort | uniq -c | sort -rn | head -20

# 共变分析（经常一起变更的目录 = 高耦合）
git log --since="6 months ago" --name-only --pretty=format:"---" | awk 'BEGIN{RS="---"} NF>1 {for(i=1;i<=NF;i++) for(j=i+1;j<=NF;j++) print $i, $j}' | sed 's|/[^/]*$||g' | sort | uniq -c | sort -rn | head -15

# 文件类别计数
echo "=== 组件 ===" && find src/visual/components -name "*.tsx" 2>/dev/null | wc -l
echo "=== API 路由 ===" && find app/api -name "route.ts" 2>/dev/null | wc -l
echo "=== 脚本 ===" && find scripts -name "*.mjs" 2>/dev/null | wc -l
echo "=== 测试 ===" && find tests -name "*.test.*" 2>/dev/null | wc -l
```

### 步骤 1-2：分析与分组

从数据推断自然领域边界：

- 变更份额 >5% 的目录 → 候选独立领域
- 频繁共变的目录 → 同一 agent
- 很少共变的目录 → 可能拆分

**耦合规则**：如果 A 经常变更，B 通常也变吗？是 → 同一 agent。否 → 可能拆分。

### 步骤 2.5：用户确认

使用运行时支持的任何提示/确认机制：

- 列出每个候选 agent：名称、职责领域、证据
- 询问分组是否匹配用户的心理模型
- **铁律**：如果用户说"这两种能力类型不同"，即使数据显示耦合也要拆分

## 阶段 2：按需设计

### 站选择

**Genesis（灵魂）和 Artisan（技能）为每个 agent 运行。** **Scout**、**Sentinel** 和 **Librarian** 是条件工厂站。**Conductor** 仍在工厂之外，仅拥有编排。

步骤 3（Genesis）后，为每个 agent 回答：

| 问题 | 是 → 站 | 理由 |
|----------|---------------|-----------|
| 本地能力覆盖是否缺失？ | Scout | 仅在基线证明缺口真实后搜索外部技能/工具 |
| 它会修改文件、调用外部 API 或触及数据库吗？ | Sentinel | 写入 = 风险面 |
| 它需要记住之前的工作或积累学习吗？ | Librarian | 跨会话一致性 |
| 它需要交接给其他 agent 或协调顺序吗？ | Conductor | 多 agent 协作 |

所有工厂问题为否 → 仅 Genesis + Artisan。如果 Conductor 问题为是，在工厂卡完成后打开编排设计。

### 步骤 3：Genesis —— 灵魂设计（必需）

**读取** `.claude/agents/meta-genesis.md` 并按该方法设计 SOUL.md。

参见 references/meta-theory.md，模块 8。

### 步骤 4：Artisan —— 技能匹配（必需）

**读取** `.claude/agents/meta-artisan.md`

1. **扫描技能**：`ls .claude/skills/*/SKILL.md` + 内置技能
2. **ROI 评分**：`ROI = (任务覆盖 × 频率) / (上下文成本 + 学习曲线)`
3. **输出**：每个 agent 的技能短名单（前 5-8 个）及 ROI 和理由

### 步骤 5：Sentinel —— 安全设计（按需）

**读取** `.claude/agents/meta-sentinel.md`

- **威胁模型**：该 agent 领域的前 5 威胁
- **权限**：三级（CAN / CANNOT / NEVER）
- **Hook**：PreToolUse / PostToolUse / Stop
- **输出**：安全规则 + Hook 配置 + 权限边界

### 步骤 6：Librarian —— 记忆设计（按需）

**读取** `.claude/agents/meta-librarian.md`

- **记忆架构**：三层（索引/主题/归档）
- **过期**：按类型保留规则
- **输出**：MEMORY.md 模板 + 持久化策略

### 步骤 7：Conductor —— 编排设计（按需）

**读取** `.claude/agents/meta-conductor.md`

- **协作流**：agent 间调用顺序、并行 vs 串行
- **触发**：何时派生此 agent
- **输出**：工作流配置 + 触发规则

## 阶段 3：审查与修订

参见 references/meta-theory.md 第 4-5 节（质量评级 + AI-slop 检测）。

### 步骤 8：关键审查

#### 8a. 自我批判

为每个 agent 的完整设计回答：

1. **我假设了什么？有证据吗？**
2. **如果我重命名 agent，设计仍然成立吗？**
3. **有任何"方便"的捷径吗？**
4. **什么是真正经过思考的 vs 模板化的？**

### 步骤 9：修订

- **B**：添加具体案例、数据引用、文件路径
- **C**：用项目特定数据重写泛化段落
- **D**：从零开始重新运行相关站

重新进入步骤 8 直到 **A 或更好**。**最多 2 轮**

### 步骤 11：最终验证

| 检查 | 方法 | 失败时 |
|-------|--------|---------|
| 五标准 | 每个 agent 一张表，5/5 通过 | 回到步骤 9 |
| 四种死法 | 无"大杂烩"、无"碎片化" | 回到步骤 2 重新分组 |
| 八模块 SOUL | 全部八个存在 | 回到步骤 3 |
| 跳过理由 | 每个跳过都有解释 | 无 → 运行该站 |

### 步骤 12：用户签收

展示完整摘要：

- 每个 agent 的角色 + 质量等级（S/A/B）
- 跳过的站及原因
- 五标准表

**仅在用户显式确认后写入文件。**
