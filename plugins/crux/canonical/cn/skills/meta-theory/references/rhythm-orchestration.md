# 节奏编排 —— 完整参考

> 精炼入规范元理论参考集；与 Meta_Kim 方法论对齐。

## 核心主张

**成熟的系统必须拆分、组合、治理——还要出牌。**

编排不仅是排序（谁先走）。它包括**节奏**：何时交付什么——以及何时**不**交付。

---

## 注意力成本模型

### 定律 1：出牌有成本

> **每次给用户的消息都消耗注意力。**

| 场景 | 成本 |
|----------|------|
| 一个建议 | 占用用户注意力带宽 |
| 一个新待办 | 与现有待办竞争 |
| 一个额外的 agent 目标 | 稀释当前焦点 |
| 一个推送的任务 | 增加认知负载 |

每条额外消息都与之前的消息竞争。注意力、带宽和吞吐量是有限的。

### 定律 2：时机改变价值

> **成熟的系统不说它们知道的一切——它们在最重要的时候说最重要的事。**

同一信息，不同时刻 → 完全不同的价值：

- 用户执行时的安全警告 → 高价值（防止伤害）
- 用户在消化上次输出时的新任务 → 低价值（过载）
- 完成一轮后的进化建议 → 中价值（有空间吸收）

### 定律 3：沉默即设计

> **什么都不做不总是失败。有时它是最优行动。**

## 事件牌组

### 设计理念

来自开放世界任务设计：**表面自由，隐藏理想节奏。**

用户感觉自由；最优交付顺序是设计的。触点在当前应出现时呈现内容。

### 牌 Schema

```yaml
card:
  id: string           # 如 "guide-01"
  type: enum           # clarify / shrink-scope / options / execute / verify / fix / rollback / risk / nudge / pause
  priority: 1-10       # 默认优先级（10 = 最高）
  cost: low|mid|high   # 注意力成本层级
  precondition: string # 如 "需求清晰"
  skip_condition: string # 如 "用户已经知道"
  interrupt_trigger: string # 抢占条件
  delivery_shell: string   # 参见意图放大
  max_iterations: number   # 迭代牌（默认 3）
```

### 十张牌（与规范理论参考对齐）

| 牌 | 原名 | 触发 | 行动 | 注意力 | 哲学 |
|------|---------------|---------|--------|-----------|------------|
| **Clarify** | 澄清 | 需求模糊 | ≤2 轮提问 | 低 | 行动前收集 |
| **Shrink scope** | 范围收缩 | 仓库太大/文件多/冲突 | 缩小边界 | 低 | 高复杂度 → 鲁莽编辑失败 |
| **Options** | 方案 | 需求清晰、多条路径 | 路线 + 权衡 + 选择 | 中 | 写代码前先有地图 |
| **Execute** | 执行 | 计划完成、风险可接受 | 分配 meta、修改代码 | 高 | 代码前先有计 |
| **Verify** | 校验 | 执行完成 | 构建/类型/依赖/需求 | 中 | 第一遍 ≠ 正确 |
| **Fix** | 修复 | 校验失败 | 修复直到通过 | 中 | 限制迭代次数 |
| **Rollback** | 回滚 | 风险或影响范围增长 | 最后稳定状态 | 高 | 后退 = 成熟 |
| **Risk** | 风险 | 共享组件/认证/全局/多方 | 暴露风险；预防 | 高 | 安全/全局优先 |
| **Nudge** | 建议 | 用户卡住、轻微触碰 | 低成本的下一步 | 低 | 有帮助但不吵闹 |
| **Pause** | 留白 | 连续完成/消化/≥3 张高成本 | 停止推送；简短状态 | 零 | 设计的沉默 |

### 出牌规则（优先级顺序）

1. **默认**：按 `priority` 出牌
2. **每张牌后**：评估下一张的 `skip_condition`——如果为真，跳过
3. **连续 ≥3 张 `high` 成本后**：强制 **Pause**
4. **如果 `interrupt_trigger` 触发**：抢占牌到最前面
5. **迭代牌**：最多 `max_iterations`；否则上报 Warden

### 出牌流程

```
[当前牌完成]
  ↓
检查下一张 skip_condition
  ├─ 满足 → 跳过，继续
  └─ 不满足 → 检查中断队列
       ├─ 抢占 → 移到最前
       └─ 无抢占 → 检查暂停规则
            ├─ 连续 ≥3 张高成本 → 强制 Pause
            └─ 否则 → 出下一张
```

## 七条启发式（开放世界 → AI 系统）

### 1. 表面自由，底层理想顺序

编排 meta 不仅决定顺序，还决定**何时**说话、**何时**沉默、什么先后、何时抢占、何时跳过。

**应用**：编排成为节奏控制，不仅是任务顺序。

### 2. 出牌接口

NPC、布告板、营地是**交付接口**，而非内容本身。

**应用**：聊天、通知、仪表板、agent 回复是出牌者。有些 meta 交付而非创作。

| 通道 | 何时 | 注意力 |
|---------|------|-----------|
| 直接回复 | 实时交互、即时反馈 | 高 |
| 写文件 | 大型制品、持久化、异步读取 | 低 |
| 派生子 agent | 专业工作 | 中 |
| 等待用户 | 需要输入/决策 | 零（等待中） |
| 通知/摘要 | 后台工作、状态 | 低 |

### 3. 暂停机制

暂停不推进工作；它**减少噪音**、防止过载、留出消化空间、保留探索感。

**触发条件**：

- 连续 ≥3 次密集推送
- 用户未回复上次输出
- 信息密度超过消化阈值

**暂停期间**：

- 无新任务
- 简短状态（"进度：X/Y 完成"）
- 等待用户驱动下一步

### 4. 成本感知出牌

每次推送都与之前的推送竞争。

**规则**：

- 推送前：现在推送比五分钟后更有价值吗？
- 否 → 不要推送
- 不确定 → 降级为写文件（更低成本）

### 5. 跳过机制

跳过是注意力管理。

**跳过当**：

- 用户已经知道
- 上下文已经包含
- 密集连续后预算耗尽 → 跳过或降级

### 6. 紧急治理

某些信号可以覆盖默认节奏。

| 来源 | 信号 | 抢占类型 |
|--------|--------|--------------|
| Sentinel | 安全/权限问题 | 安全——最高 |
| Prism | 严重质量漂移（如 pass_rate < 0.5） | 质量 |
| 用户 | "紧急"/"现在" | 用户 |
| 系统 | 资源/超时/错误 | 系统 |

**Sentinel → Conductor**：`{type: "interrupt", source: "sentinel", severity: "critical", detail: "..."}` → 暂停牌组，安全牌前置。

**Prism → Conductor**：`{type: "interrupt", source: "prism", severity: "high", detail: "..."}` → 现在 critical；高在下一张牌之前。

### 7. 同一意图，多种 Shell

核心稳定；Shell 按场景切换。

**应用**：每张牌上，选择出**什么**和**哪个 Shell**（意图放大）。

### Conductor

Conductor **执行**节奏；本文档是**方法**。实现（牌数据、出牌函数、暂停/抢占）位于 `.claude/agents/meta-conductor.md`。

暂停和冲突的上报：

- 迭代超过 `max_iterations` → Warden 决定
- 冲突抢占 → Warden 仲裁

**发送**抢占信号；Conductor 接收并应用。

## 场景

### 1. 快乐路径

```
Clarify(low) → Shrink(low) → Options(mid) → Execute(high) → Verify(mid)
→ Fix(mid) → [进化扫描] → 完成
```

### 2. 过载 → 暂停

```
Execute(high) → Verify(high) → Fix-1(high)
→ [3× high] → 强制 Pause → "修订轮次 2：1/3 检查通过"
→ 用户继续 → Fix-2(mid)
```

### 3. 紧急抢占

```
Execute(high) → [Sentinel 警报] → 暂停牌组
→ Risk（最高）→ 修复安全问题 → 恢复 → Verify(mid)
```

### 4. 回滚

```
Execute(high) → [范围爆炸] → Rollback(high)
→ 稳定状态 → 重新评估 → Shrink(low) → 重新 Execute
```
