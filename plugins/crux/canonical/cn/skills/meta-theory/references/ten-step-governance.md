# 十步治理 —— 完整参考

> 精炼入规范元理论参考集；与 Meta_Kim 方法论对齐。
> **步骤 1-3 = "动起来了"。步骤 4-10 = "成熟了"。**

## 核心主张

治理意味着：**系统不仅做工作——还判断、纠正、验证和进化。**

第一次输出只证明运动。运动 ≠ 可靠性 ≠ 可复用性 ≠ 可进化性。

> **AI 成熟度不是"第一稿有多好"，而是是否存在自我纠正循环。**

---

## 十步地图

```
方向(1) → 计划(2) → 执行(3) → 审查(4) → 元审查(5) → 修订(6) → 验证(7) → 摘要(8) → 反馈(9) → 进化(10)
├────── 动起来了 ──────┤├──────────────────── 成熟了 ────────────────────────────────────────┤
```

## 逐步详解

### 步骤 1：方向

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | Warden + 用户 |
| **输入** | 用户需求（可能模糊） |
| **核心工作** | 清晰度检查 → 意图核心 → 复杂度分类 |
| **输出** | 清晰的意图核心 + 复杂度（简单/中等/复杂） |
| **门控** | 核心必须可测试：能否用一句话描述"完成的样子"？如果不能 → 澄清 |

**澄清规则**：

- 最多 2 轮（引导牌）
- 2 轮后仍模糊 → 保守假设，标记假设
- 不要永远猜测意图

### 步骤 2：规划

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | Conductor |
| **输入** | 意图核心 + 复杂度 |
| **核心工作** | 分解 → 分配 meta → 依赖 → 牌组 |
| **输出** | 任务计划 + 牌组配置 + 并行/串行标志 |
| **门控** | 每个子任务映射到具体 meta；如果没有 → 创建流水线 |

**此处的复杂度路由**：

- 简单（<2 个文件）→ 跳过重度规划，执行
- 中等（2-5 个）→ 轻量计划
- 复杂（>5 个）→ 完整计划 + 依赖图

### 步骤 3：执行

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | 执行 meta（按分配） |
| **输入** | 计划 + 具体子任务 |
| **核心工作** | 搜索资产 → 匹配能力 → 执行 → 制品 |
| **输出** | 代码/文档/配置/设计 |
| **门控** | 每个文件同时只有一个写入者 |

**执行规则**：

- 无重叠文件 → 安全并行
- 有重叠 → 串行；第二个等待
- 无拥有 meta → Type B 创建流水线

### 步骤 4：审查

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | Prism |
| **输入** | 步骤 3 制品 |
| **核心工作** | Slop 扫描 → 断言审查 → 声明检查 → 深度 → 评级 |
| **输出** | Prism 报告：断言通过/失败 + 评级 S/A/B/C/D |
| **门控** | A/S 通过；B 修订；C/D 重做 |

**审查不是一个分数**——它是取证的：

- 每个声明需要证据（路径、行号、数据）
- 必须暴露并测试隐藏假设
- 无证据 → 失败

### 步骤 5：元审查

> **谁审查审查者？**

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | Warden（审查 Prism 的**标准**，而非重新判断制品） |
| **输入** | Prism 报告 + 审查标准 |
| **核心工作** | 断言覆盖 → 断言强度 → 标准一致性 |
| **输出** | 通过/需要更多/漂移警告 |
| **门控** | 参见下方元审查协议 |

#### 元审查协议

Warden 审查 Prism 的标准，而非制品本身：

| 维度 | 方法 | 失败时 |
|-----------|--------|---------|
| **覆盖** | 断言是否覆盖所有关键维度？ | 添加缺失维度 |
| **强度** | 弱断言制造虚假信心？ | 收紧（如"有 Core Truths" → "≥3 条并通过可替换性测试"） |
| **一致性** | 与上次同类审查标准相同吗？ | 记录差异：进化 vs 漂移 |

**触发条件**：

```
如果 Prism 通过率 > 0.9 且输出明显差
  → 强制元审查（标准太松）

如果 Prism 通过率 < 0.3 且输出看起来合理
  → 强制元审查（标准太严）

如果标准与上次同类审查差异 >30%
  → 漂移警告 → Warden 决定。
```

> **弱的通过比失败更糟——它制造虚假信心。**

### 步骤 6：修订

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | 原始执行 meta |
| **输入** | Prism 报告 + Warden 元审查说明 |
| **核心工作** | 按反馈修复 → 重新提交 |
| **输出** | 修订后的制品 |
| **门控** | 最多 3 轮 AutoFix；3 轮后仍 B → Warden。AutoFix = 无需用户的自动修复循环直到上限。 |

**修订层级**：

- B → 添加案例、引用
- C → 用真实数据替换泛化文本
- D → 从头重新执行

### 步骤 7：验证

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | Prism（独立于修订者） |
| **输入** | 修订后的制品 |
| **核心工作** | 重新运行相同断言 → 确认修复已落地 |
| **输出** | 通过/需要更多修订 |
| **门控** | 需要新证据——不是"我觉得我修好了" |

> **验证不是自我庆祝。改变 ≠ 修复。重新运行断言。**

### 步骤 8：摘要

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | Warden |
| **输入** | 完整运行数据 + 最终制品 |
| **核心工作** | 整合报告 → 提取学习 → 面向高管的备忘录 |
| **输出** | 高管备忘录（Shell 匹配）+ 学习日志 |
| **门控** | 通过备忘录的意图放大检查 |

**摘要 ≠ 粘贴**——调和冲突、为步骤 10 提取学习。

### 步骤 9：反馈

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | 用户/高管 |
| **输入** | 摘要 + 制品 |
| **核心工作** | 满意度 → 变更请求 → 接受 |
| **输出** | 确认/请求变更 |
| **门控** | 显式确认；模糊的（"大概还行"）→ 询问哪里不行 |

### 步骤 10：进化

| 属性 | 内容 |
|-----------|---------|
| **拥有者** | 所有人（每个 meta）+ Conductor 协调 |
| **输入** | 完整追踪 + 用户反馈 |
| **核心工作** | 五维扫描 → 放大行动 |
| **输出** | 进化待办 + 已执行的变更 |
| **门控** | 至少一个维度有具体行动（不能全是"无变更"） |

详情：`references/intent-amplification.md`。

## 复杂度路由

不是每个任务都运行全部十一步。

### 简单（<2 个文件）

```
方向(1) → 执行(3) → 审查(4) → 验证(7) → 反馈(9)
跳过：计划(2)、元审查(5)、修订(6)、摘要(8)、进化(10)
```

小变更：审查 + 验证即可；不需要元审查或完整进化。

### 中等（2-5 个文件）

```
方向(1) → 计划(2) → 执行(3) → 审查(4) → 元审查(5) → 修订(6) → 验证(7) → 反馈(9)
跳过：摘要(8)、进化(10)
```

规划 + 元审查在此规模很重要；进化可选。

### 复杂（>5 个文件/多模块）

```
完整 1-10
```

大变更：元审查防止盲点，摘要，进化。

### 升级矩阵

| 信号 | 升级 |
|--------|----------|
| 文件数超过 2 | 简单 → 中等 |
| 跨模块依赖 | 中等 → 复杂 |
| Sentinel 安全 | 任何 → 复杂（完整路径） |
| Prism 发现系统性问题 | 中等 → 复杂 |
| 用户要求完整流程 | 任何 → 复杂 |

## 信号流

```
方向(1) ──核心+复杂度──→ 计划(2)
计划(2) ──牌组+分配──→ 执行(3)
执行(3) ──制品──→ 审查(4)
审查(4) ──报告──→ 元审查(5)
         ──反馈──→ 修订(6)
元审查(5) ──标准修复──→ 修订(6)（如需要）
修订(6) ──修订后──→ 验证(7)
验证(7) ──通过──→ 摘要(8)
         ──失败──→ 修订(6)（循环，最多 3 轮 AutoFix）
摘要(8) ──高管备忘录──→ 反馈(9)
反馈(9) ──确认──→ 进化(10)
         ──变更──→ 修订(6)
进化(10) ──行动──→ [下次运行更强]
```

## 元审查深入分析

最容易被跳过；跳过后果最严重。

### 为什么存在

Prism 判断输出质量。**谁判断 Prism 的判断？**

**风险**：

- 太松 → 差的输出通过（虚假信心）
- 太严 → 好的输出被阻止（浪费）
- 漂移的标准 → 同样的工作今天通过、明天失败

### 元审查**不做**什么

- 重新审查制品（Prism 的工作）
- 挑剔 Prism 的结论（除非标准本身有问题）
- 替代 Prism

### 元审查**做**什么

- 关键维度的断言覆盖
- 断言强度（弱通过检测）
- 历史一致性（漂移）

### Prism 在元审查下的义务

- 发布完整断言列表和评分标准
- 解释每个断言的设计
- 标记与上次同类审查的差异
- 接受 Warden 的标准调整

参见 `.claude/agents/meta-prism.md` "被审查方协议"。
