端到端的元方法论：

```
Meta（拆分） → 组织镜像（组合） → 节奏编排（出牌） → 意图放大（交付）
```

| 层 | 解决什么 | 核心问题 |
|-------|----------------|---------------|
| **Meta** | 如何拆分 | 系统应该被分解到什么粒度才能既工作又可治理？ |
| **组织镜像** | 如何组合 | meta 如何形成结构化的组织？ |
| **节奏编排** | 如何出牌 | 组织如何优雅地出牌而非一次性倾倒？ |

## Meta = 最小可治理单元

### 五标准

**Meta = 最小可治理单元**必须满足：

1. **独立**——可独立理解、调用和产出
2. **足够小**——进一步拆分无意义或成本高于收益
3. **边界清晰**——显式的"Own"与"Do Not Touch"
4. **可替换**——替换不会崩溃；可升级或重组
5. **可复用**——跨场景有用，不是一次性的

五标准证据表（审计时使用）：

| 标准 | 证据 | 通过 |
|-----------|----------|------|
| 独立 | 可观察的调用 + 输出 | 是 / 否 |
| 足够小 | 拆分成本 vs 收益 | 是 / 否 |
| 边界清晰 | 书面 owns / does-not-touch | 是 / 否 |
| 可替换 | 替换实验笔记 | 是 / 否 |
| 可复用 | 跨场景复用证明 | 是 / 否 |

### 四种死法

| 死法 | 症状 | 诊断问题 |
|---------------|----------|----------------------|
| **大杂烩（Stew-All）** | 一个 agent 做所有事 | >2 个不相关领域？SOUL.md >300 行？ |
| **碎片化（Shattered）** | 太多微小 agent | 需要其他 agent 的输出才能产出任何东西？ |
| **无治理执行（Governance-Free Execution）** | 只有方向→计划→执行；没有审查→验证→进化 | 谁审查？谁审查审查者？谁验证修复？学习如何被捕获？ |
| **追结果无结构（Result-Chasing Without Structure）** | "跑过一次"被当作真理 | 明天还能跑吗？交接能工作吗？另一个模型能适应吗？ |

> 死法 1-2 是拆分极端；3-4 是治理缺口。
> 第一对使系统不稳定；第二对阻止系统成熟。

### 全能执行者 Meta 反模式

**全能执行者 Meta 反模式**——一个 meta 吸收了执行、审查和合成，直到系统表现出**压缩病**（一切都坍缩为一个过载角色）。补救措施：**触发 Type B 拆分**，让 Genesis/Artisan 站重建边界。

需要标记的症状：

- **在充分理解之前执行**
- **在完整信息收集之前做决策**
- **在暴露风险之前修改共享逻辑**

**大杂烩诊断**（与上表对齐）：注意一个 SOUL 中的 **>2 个不相关领域**和 **>300 行** SOUL.md 但没有拆分计划。

**铁律**（拆分）：如果**用户说**"这两个**关注点不同**；**把它们分开"，即使 git 数据显示高耦合也视为强制拆分——用户在耦合决策上的**覆盖是最终的**。

**耦合探针**（数据 + 判断）：**如果 A 变了，B 是否经常也需要变？**高**共变频率** → 候选**合并**；低**共变频率** → 领域可以保持**分离**。

通俗合并规则（用于搜索/正则检查）：高共变频率应该合并；低共变频率可以分离。

**元验证四问**（在拆分提案后）：

1. **owns / does-not-touch 之间有清晰边界吗？**
2. **替换它会导致相邻 meta 崩溃吗？**
3. **跨污染在哪里仍可能泄漏？**
4. **这个 meta 能与其他 meta 组合而不发生角色崩溃吗？**

### 拆分的黄金带

**足够小又足够完整；独立又能协作；负责而不拖垮整个系统。**

启发式：

- 进一步拆分 → 治理成本占主导 → 下界
- 向上合并 → 职责模糊 → 上界
- 之间的区间是黄金间隔

### 三个层级

| 层级 | 角色 | 示例 |
|------|------|----------|
| 执行 meta | 做事；保持范围纯粹 | 前端、后端、质量 |
| 编排 meta | 调度工作、排序 | Commander、meta-conductor |
| 基础设施 meta | 构建能力，而非任务 | 记忆/技能/工具/规则/权限/安全 |

> **有些 meta 不"做任务"；它们构建能力。**

执行 meta 在前线战斗。编排 meta 运行作战地图。基础设施 meta 建造机场、道路、弹药和规则。

## 入口 Meta

### 定义

**每个项目有且仅有一个入口 meta。**

入口 meta = 用户对话的第一个 agent = 谁将意图转化为可执行的工作包。

它不是"创建顺序"；它是**入口集中**——对用户一个声音，而非轮换角色。

### 入口 Meta 与其他 Meta

| 维度 | 入口 Meta | 其他 Meta |
|-----------|------------|-------------|
| **入口** | 用户直接对话 | 由入口派生/路由 |
| **角色** | 意图翻译 + 路由 + 节奏 | 执行具体任务 |
| **边界** | "Owns" 用户可见层 | "Owns" 内部执行 |
| **数量** | 每项目一个 | 每项目 N 个 |

### 职责（仅入口 Meta）

| 职责 | 含义 |
|----------------|---------|
| **意图翻译** | 将模糊请求转化为具体任务 |
| **任务路由** | 决定哪个执行 meta 拥有工作 |
| **节奏控制** | 何时说话，何时沉默 |
| **门控** | 阻止超范围变更（如前端编辑 API） |
| **统一摘要** | 面向用户的合成仅来自入口 meta |

### 输出

其他人"听到"的单一声音：

```
[结构化需求]
  ↓
[任务分解 + agent 路由]
  ↓
[自主执行 + 自动审查]
  ↓
[统一变更摘要]  ← 用户只看到这一层
```

### 项目规模 vs 入口名称

| 规模 | 入口名称 | 层级 | 特征 |
|-------|------------|------|--------|
| **单项目** | Commander | 项目执行 | 需求指导 + 分解 + 分发 |
| **多部门** | Warden | Meta 层 | 协调 + 仲裁 + 最终合成 |
| **通用模板** | Manager | 通用入口 | 意图 → 计划 → 交付 |

### 核心原则

1. **对用户一个声音**——执行 meta 不要越过入口 meta 说话
2. **入口 Meta 不写代码**——输出是计划和调度，不是实现
3. **入口 Meta 是守门员**——超范围变更必须被阻止
4. **入口 Meta 设定节奏**——何时推进、暂停或中断

> **没有入口 meta = 多个 agent 同时说话 = 认知过载**

### 在组织镜像中的位置

入口 meta 是镜像的"用户界面层"：

```
用户 ──→ 入口 Meta ──→ 执行 Meta ──→ 治理 Meta ──→ 用户
            ↑
         意图翻译
         任务路由
         节奏控制
         门控
```

## 组织镜像

### 定义

不是隐喻——一种将真实组织机制映射到多 agent 系统的架构模式：

| 机制 | 在组织中 | 在 AI 系统中 |
|-----------|---------|---------------|
| 分层委托 | 上层→中层→执行 | CEO → Warden → 执行 agent |
| 角色分工 | 岗位描述 | SOUL.md + owns / does-not-touch |
| 分离工作空间 | 部门 | 每 agent 的上下文隔离 |
| 审查反馈 | QA + 绩效 | Review + Meta-Review |
| 持续改进 | 复盘 + 培训 | Evolution + 记忆 |

### 三种经典失败（没有镜像时）

| 失败 | 症状 | 根因 |
|---------|---------|------------|
| **跨污染** | 领域 A 泄漏到 B | 无隔离的共享上下文 |
| **协调爆炸** | 角色太多、链接混乱 | 扁平消息、无结构 |
| **高设计成本** | 手工制作每个交互 | 无组织形抽象 |

> **协作可以是扁平的；治理不能缺失。平等 ≠ 混乱。**

## 节奏编排（摘要）

### 定义

编排不仅是排序（谁先走）。它包括**节奏**：何时交付什么——以及何**不**交付。

### 核心理念

> **告诉用户信息有成本。**

- 每个建议竞争注意力
- 每个待办与其他待办竞争
- 每个额外目标稀释焦点
- 每个推送的任务增加认知负载

注意力、带宽和吞吐量是有限的。

> **成熟的系统不说它们知道的一切；它们在最重要的时候说最重要的事。**

### 事件牌组

牌组实现节奏：表面自由，底层理想顺序。

| 牌 | 触发 | 行动 | 哲学 |
|------|---------|--------|------------|
| **Clarify（澄清）** | 需求模糊 | ≤2 轮提问 | 行动前收集 |
| **Scope shrink（范围收缩）** | 仓库太大/文件太多/名称冲突 | 缩小到可工作的边界 | 高复杂度 → 鲁莽编辑会失败 |
| **Options（方案）** | 需求清晰、多条路径 | 列出路线、权衡、建议 | 大问题需要先有地图再写代码 |
| **Execute（执行）** | 计划完成、风险可接受 | 分配给 meta、触及代码 | 有计划地执行 |
| **Verify（校验）** | 执行完成 | 构建/类型/依赖/需求 | 第一遍 ≠ 正确 |
| **Fix（修复）** | 校验失败 | 修复直到通过；限制迭代次数 | 不要伪造"完成" |
| **Rollback（回滚）** | 风险或影响范围增长 | 返回最后稳定状态 | 后退是成熟 |
| **Risk（风险）** | 共享组件/认证/全局逻辑/多方 | 暴露风险；必要时预防 | 安全/权限/全局影响优先 |
| **Nudge（建议）** | 用户卡住、轻微触碰可接受 | 低成本的下一步 | 非紧急但有用 |
| **Pause（留白）** | 连续完成/需要消化/≥3 张高成本牌 | 停止推送；仅简短状态 | 设计的沉默可能是最优的 |

### 三个内部机制

Conductor 内部机制——不是独立的 meta（它们通不过五标准）：

| 机制 | 触发 | 行为 |
|-----------|---------|----------|
| **暂停节奏** | ≥3 张连续高成本牌 | 停止推送；简短状态 |
| **回滚** | 风险或范围超出计划 | 回退到稳定状态，重新评估 |
| **紧急治理** | Sentinel 警报 / 严重 Prism 漂移 | 暂停牌组；风险牌前置 |
| **出牌接口** | 每张牌 | 选择通道（回复/文件/子agent/等待） |

## 意图放大（摘要）

### 定义

**结构性扩展**高层意图的结果——不是一个人做所有事，而是意图通过组织层逐层放大为系统级输出。

### 意图核心 + 交付 Shell

> **一个意图不应只有一个表达。成熟的系统按触点、角色和上下文切换 Shell，而非核心。**

- **意图核心**（稳定）：目标 / 信息 / 决策
- **交付 Shell**（上下文相关）：按角色、通道、UI、格式包装

Shell 选择的四个维度：

1. **受众**——高管 / 开发者 / 最终用户
2. **触点**——文档 / 聊天 / 通知
3. **上下文密度**——首次查看 / 重访 / 紧急
4. **注意力预算**——高 / 中 / 低

> 细节：`references/intent-amplification.md`

### 五维进化扫描

| 维度 | 检查什么 | 通过标准 |
|-----------|---------------|----------|
| **模式复用** | 这个解决方案能成为可复用模式吗？ | 清晰的组件/模板/规则 |
| **Agent 边界** | 仍然有效？需要拆分/合并？ | 五标准仍然通过 |
| **引导 UX** | 路径能更短/更顺畅吗？ | 更少的澄清轮次或更简单的输入 |
| **流程瓶颈** | 最慢或最易出错的步骤？ | 已识别并附修复方案 |
| **能力覆盖** | 新缺口？ | 已记录或触发创建流水线 |

### 五项放大行动

| 维度 | 发现 | 行动 |
|-----------|---------|--------|
| 模式复用 | 可复用模式 | 提取技能/模板 → Artisan 池 |
| Agent 边界 | 不合适 | 拆分/合并 → Type B 流水线 |
| 引导 UX | 冗余路径 | 更新牌触发条件/澄清 |
| 流程瓶颈 | 瓶颈 | 重新排序牌组；并行化或跳过 |
| 能力覆盖 | 缺口 | 新 meta/技能或 Scout 搜索工具 |

> 细节：`references/intent-amplification.md`

## 十步治理（摘要）

| 步骤 | 含义 | 拥有者 | 关键问题 |
|------|---------|-------|--------------|
| 1. 方向 | 需求 | Warden / 用户 | 意图是什么？ |
| 2. 规划 | 分解 | Conductor | meta 如何拆分？ |
| 3. 执行 | 搜索 + 做 | 执行 meta | 谁来做？ |
| 4. 审查 | 质量 | Prism | 正确吗？ |
| 5. 元审查 | 审查审查者 | Warden 审查 Prism | 标准合理吗？ |
| 6. 修订 | 修复 | 执行 meta | 如何修复？ |
| 7. 验证 | 重新检查 | Prism | 真的修复了？不是自我庆祝？ |
| 8. 摘要 | 合成 | Warden | 我们学到了什么？ |
| 9. 反馈 | 用户确认 | 用户 / 高管 | 满意吗？ |
| 10. 进化 | 五个维度 | 所有人 | 下次如何更强？ |

> 步骤 1-3 = "动起来了"。4-10 = "成熟了"。
> 没有审查 → 盲目冲锋。没有验证 → 走过场。没有进化 → 重复错误。

### 复杂度路由

| 复杂度 | 规则 | 步骤 |
|------------|------|-------|
| 简单 | <2 个文件 | 1→3→4→7→9（跳过元审查） |
| 中等 | 2-5 个文件 | 1→2→3→4→5→6→7→9 |
| 复杂 | >5 个文件/多模块 | 全部 10 步 |

> 细节：`references/ten-step-governance.md`

## 八模块 SOUL 设计

| # | 模块 | 角色 | 质量标准 |
|---|--------|------|-------------|
| 1 | Core Truths | 行为锚点 | ≥3 条，领域特定 |
| 2 | 角色 + 核心工作 | 身份 + 核心职责 | 清晰的 owns / does-not-touch |
| 3 | 决策规则 | 不确定性下的选择 | ≥3 条 if/then 规则 |
| 4 | 思维框架 | 如何在领域内思考 | 领域特定步骤 |
| 5 | 反 AI-Slop | 反模板信号 | 具体的 slop 检测器 |
| 6 | 输出质量 | 可验证的标准 | 可观察的标准 |
| 7 | 交付物流 | 交接路径 | 清晰的输入→处理→输出 |
| 8 | Meta-Skills | 自我改进 | ≥2 个学习方向 |

## 质量评级

| 等级 | 标准 | 行动 |
|-------|----------|--------|
| **S** | 独到洞察、硬数据、可执行、难替代 | 通过 |
| **A** | 完整覆盖、具体数据、中等深度 | 通过 |
| **B** | 结构完整但案例/数据弱 | 修订 |
| **C** | 泛化、可替换、无真实计划 | 修订 |
| **D** | 模板倾倒、无思考痕迹 | 重做 |

## AI-Slop 检测

| 信号 | 如何检测 | 判定 |
|--------|---------------|---------|
| 模板密度 | 计算"总而言之/值得注意的是/总体来说" | >0 扣分 |
| 缺乏具体性 | 无文件/函数/数据引用 | 失败 |
| 可替换性 | 将 agent 名称替换为竞争对手名称；内容仍"成立"？ | 浅薄 |
| 列表充填 | 5+ 条建议每条 <2 句话 | 肤浅 |
