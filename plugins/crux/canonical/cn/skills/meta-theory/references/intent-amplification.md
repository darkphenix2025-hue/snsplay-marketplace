# 意图放大 —— 完整参考

> 精炼入规范元理论参考集；与 Meta_Kim 方法论对齐。

## 核心主张

**高层意图，结构性扩展。** 不是一个人做所有事——意图通过组织层逐层放大为系统级输出。

> **一个意图不应只有一个表达。成熟的系统按触点、角色和上下文切换 Shell，而非核心。**

---

## 意图核心 + 交付 Shell

### 意图核心（稳定）

底层的**目标 / 信息 / 决策**。不随包装改变。

示例：

- "认证系统需要 token 刷新机制"——意图核心
- "进度比计划落后约 20%"——意图核心
- "Genesis SOUL.md 缺少决策规则"——意图核心

### 交付 Shell（上下文相关）

核心如何为给定受众包装。同一核心，不同 Shell。

示例——核心："Token 刷新必须实现"：

| 受众 | Shell |
|----------|-------|
| 高管 | "认证模块缺少一个安全关键项；大约需要额外 2 天。" |
| 开发者 | "实现 `refreshToken()`，在 JWT 过期前 5 分钟触发刷新；参见 API 设计文档。" |
| 最终用户 | "你无需重新输入凭证即可保持登录状态。" |

## Shell 选择的四个维度

### 矩阵

| 维度 | 选项 | 效果 |
|-----------|---------|--------|
| **受众** | 高管 / 开发者 / 用户 / 审计员 | 抽象层级和深度 |
| **触点** | 文档 / 聊天 / 通知 / 报告 | 格式和长度 |
| **上下文密度** | 首次查看 / 重访 / 紧急 | 多少背景信息 |
| **注意力预算** | 高 / 中 / 低 | 信息密度 |

### 决策草图

```
selectDeliveryShell(card, audience, context):

  IF audience = 高管:
    → 高抽象、结论先行、建议行动
    → 格式：摘要 + 关键数字 + 下一步

  IF audience = 开发者:
    → 低抽象、实现细节、代码引用
    → 格式：技术说明 + 路径 + 代码片段

  IF audience = 审计员:
    → 中等抽象、证据链、可验证声明
    → 格式：断言 + 证据 + 裁定

  THEN 按上下文密度调整:
    IF 首次查看 → 添加背景
    IF 重访 → 仅变更
    IF 紧急 → 结论 + 行动

  THEN 按注意力预算调整:
    IF 高 → 完整细节
    IF 中 → 核心 + 可展开链接
    IF 低 → 一行摘要
```

## 牌成本 vs Shell

Shell 选择直接影响注意力成本：

| Shell 类型 | 注意力成本 | 何时 |
|------------|----------------|------|
| 一行摘要 | 低 | 低预算、确认 |
| 结构化报告 | 中 | 中等预算、概览 |
| 完整技术文档 | 高 | 首次深入 |
| 仅变更 | 低 | 重访、仅变更 |
| 决策备忘 | 中 | 高管、需要行动 |

**与节奏的关系**：每张牌上，Conductor 为受众 + 上下文选择 Shell 以控制成本。

## 五项进化放大行动

进化扫描不是终态——它们必须成为结构性升级。

### 1. 模式复用 → 提取

| 发现 | 行动 | 拥有者 |
|---------|--------|-------|
| 可复用代码模式 | 技能 → Artisan 池 | Artisan |
| 可复用工作流 | 编排模板 → Conductor 库 | Conductor |
| 可复用审查模式 | 断言模板 → Prism 库 | Prism |

### 2. Agent 边界 → 重构

| 发现 | 行动 | 拥有者 |
|---------|--------|-------|
| 角色蔓延（>2 个领域） | 拆分 → Type B 流水线 | Warden |
| 过度碎片化 | 合并/重新分组 | Warden |
| 五标准仍然通过 | 快照验证 | Prism |

### 3. 引导 UX → 交互

| 发现 | 行动 | 拥有者 |
|---------|--------|-------|
| >2 轮澄清 | 调整引导牌/意图预设 | Conductor |
| 用户必须提供过多上下文 | 更强的记忆/历史填充 | Librarian |
| 误解率 > 0 | 收紧决策规则/消歧 | Genesis |

### 4. 流程瓶颈 → 吞吐量

| 发现 | 行动 | 拥有者 |
|---------|--------|-------|
| 最慢的步骤 | 根因 → 并行化或预计算 | Conductor |
| 最高失败率 | 预检查或更简单的回退 | Sentinel |
| 串行浪费 | 标记可并行步骤、更新牌组依赖 | Conductor |

### 5. 能力覆盖 → 规模

| 发现 | 行动 | 拥有者 |
|---------|--------|-------|
| 新能力缺口 | 新 meta/技能或 Scout | Scout / Genesis |
| 缺少工具/技能 | Scout 扫描 → ROI → 采用？ | Scout |
| 知识缺口 | 记忆/参考 + Librarian 索引 | Librarian |

## Warden：意图放大审查

在合成期间，检查交付质量：

### 高管 Shell 检查清单

| 检查 | 方法 | 失败时 |
|-------|--------|---------|
| 抽象层级正确吗？ | 高管备忘录不应嵌入原始代码路径 | 更高抽象重写 |
| 结论先行吗？ | 第一段陈述核心裁定 | 重组结构 |
| 有可操作建议吗？ | 不是仅提供信息 | 添加"建议行动" |
| 密度匹配预算吗？ | 高管通常是"中"预算 | 裁剪细节 |

### 跨受众一致性

同一核心跨受众：

- 事实必须一致（不能告诉高管"进展正常"同时告诉开发者"进度延迟"）
- 仅 Shell 不同，内容不能矛盾
- 如果矛盾 → 核对核心，然后重新 Shell

## 与其他线索的关系

```
Meta（拆分）
  ↓ 产生独立可交付的单元
组织镜像（组合）
  ↓ 产生分层受众（高管/中层/执行）
节奏编排（出牌）
  ↓ 产生时机和注意力预算
意图放大（交付）← 此层
  ↓ 产生按受众、触点、上下文的结构化交付
```

意图放大是**终端层**——上游工作在此成为有效的系统输出。
